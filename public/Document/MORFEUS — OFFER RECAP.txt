4) Delivery currency: Slot + Taglie (S/M/L) definitive
4.1 Unità: Slot
1 slot = lavoro completo “end-to-end” per un pezzo di delivery: design breve + build + QA + deploy + doc/handover minimo.
4.2 Taglie
SMALL (S) = 1 slot
Quando è S (deve rispettare TUTTO):
               * max 1–2 integrazioni (API già note e documentate)

               * max 1 workflow lineare (no branching complesso)

               * max 1 step “AI/agent” semplice (es. classificazione, estrazione) con prompt controllato

               * no UI dedicata (al massimo form/field esistenti)

               * rischio basso: no impatto su billing/core money flow

               * test: smoke test + logging base

Esempi S:
                  * sync bidirezionale semplice tra CRM e sheet/DB

                  * bot che genera summary e aggiorna record

                  * automazione di routing lead + notifiche

MEDIUM (M) = 2 slot
Quando è M (basta 1 di questi):
                     * 3–4 integrazioni oppure 2 integrazioni ma con mapping dati non banale

                     * workflow con branching (if/else), retry, idempotenza

                     * agent con tool (es. ricerca knowledge base, chiamate API multiple) + guardrail base

                     * include minima UI interna (es. dashboard light / pagina admin / approvazioni)

                     * include requisiti “non funzionali” base: rate limits, error handling strutturato, audit trail

                     * test: happy path + 3–5 edge cases

Esempi M:
                        * agent di triage ticket con escalation e SLA

                        * orchestrazione multi-step (CRM → ERP → ticketing) con controlli

                        * dashboard KPI “decision-grade” v1 se i dati sono accessibili

LARGE (L) = 4 slot
Quando è L (basta 1 di questi):
                           * 5+ integrazioni o sistemi enterprise (ERP/legacy) con vincoli

                           * componente dati/visibilità serio (model, quality checks, alerting)

                           * UI interna più completa (ruoli, approvazioni, audit)

                           * agent “multi-tool” con policy, rate limit, fallback, logging avanzato

                           * impatto su processi critici (fatturazione, ordini, margini, cash collection) → richiede gating/release safe

                           * test: suite più robusta + staging + feature flags (quando serve)

Esempi L:
                              * control surface completa + alerting + pipeline

                              * agent operativo che esegue azioni su sistemi core con approval flow

                              * refactor parziale di un processo end-to-end (senza replatform totale)

XL = Custom/Enterprise
Migrazioni importanti, rewrites, ERP deep, compliance pesante, multi-country: custom scope.

7) Contratto: annual lock + way-out “data-driven”
7.1 Struttura
                                       * Firmate contratto annuale (12 mesi).

                                       * Pagamento mensile (default). Opzione trimestrale anticipata (facoltativa).

7.2 Way-out 30 giorni (OGGETTIVO)
Non è “se mi piace”. È “se funziona come promesso”.
Regola:
                                          * Entro 30 giorni dalla Ready Date, Morfeus consegna il Pilot Build con criteri di accettazione misurabili.

                                          * Se il Pilot Build NON soddisfa i criteri concordati → il cliente può esercitare way-out e terminare il contratto (senza penali future).

                                          * Se il Pilot Build SÌ soddisfa i criteri → il contratto prosegue per l’anno e si rinnova automaticamente (salvo disdetta nei termini).

Pilot Build per tier
                                             * Tier 1: Pilot = S

                                             * Tier 2: Pilot = M

                                             * Tier 3: Pilot = M o L (scelto insieme in base all’impatto)

Criteri di accettazione (template)
Ogni Pilot Build deve avere:
                                                * “fa X” (funzionale)

                                                * tempi/affidabilità minimi (es. error rate sotto soglia)

                                                * logging/audit base

                                                * handover (doc + runbook) + demo

Questa impostazione elimina il rischio “progetto a sconto”: o consegnate “come da spec” e continuano, oppure non avete mantenuto la promessa e il cliente può uscire.

9) Il punto che hai citato: ogni deployment costa mantenerlo (policy inclusa)
Questa è fondamentale e vi salva i margini.
Maintenance Tax (regola semplice e onesta)
Ogni mese, una parte della capacità serve per:
                                                   * bugfix, monitoraggio, aggiornamenti, drift, piccole modifiche richieste dai sistemi già deployati.

Policy consigliata:
                                                      * 20% degli slot mensili sono implicitamente allocati a Maintenance & Reliability.

                                                      * Il restante 80% va su nuove build.

Esempio:
                                                         * Tier 2 = 4 slot → 0.8 slot “maintenance budget” (accorpato nella pianificazione) + 3.2 slot nuove build (in pratica: gestite con buon senso nel planning).

Warranty window (per evitare discussioni)
                                                            * Bugfix relativi a un rilascio: inclusi per 14 giorni dal deploy (se non sono change request).

                                                            * Dopo: rientrano in maintenance budget/slot.

Questo vi evita il “supporto infinito” che uccide margini.
